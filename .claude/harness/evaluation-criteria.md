# Tika 프로젝트 품질 평가 기준

이 문서는 **Evaluator** 에이전트가 Generator의 결과물을 채점할 때 사용하는 공식 기준이다.

## ✅ MUST PASS (하나라도 실패하면 스프린트 FAIL)

### 1. Regression Prevention (회귀 방지) ★★★★★
- [ ] **변경 범위 제어**: 신규 요구사항과 무관한 기존 파일/함수/동작을 변경하지 않았는가?
- [ ] **계약 준수**: 변경 파일 목록이 Sprint Contract의 허용 범위 안에 있는가? (계약 외 변경 시 즉시 FAIL)
- [ ] **Core Invariants 보존**: `MEMORY.md`에 정의된 핵심 불변 규칙들이 유지되는가?
- [ ] **Diff 감사**: `git diff --stat` 및 `git diff --name-only`를 통해 불필요한 코드 변경이 없음을 확인했는가?
- [ ] **테스트 보존**: 기존 테스트 전체가 통과하며, 테스트가 없던 중요 동작에 대해 최소 1개 이상의 보존 테스트를 추가했는가?

### 2. 기능 완성도 ★★★★★
- [ ] 스펙(`.kiro/specs/`)에 정의된 기능이 end-to-end로 동작하는가?
- [ ] stub 함수나 `// TODO`, `# TODO` 주석이 남아있지 않은가?
- [ ] 에러 케이스(boto3 ClientError, 네트워크 오류, 빈 데이터)를 적절히 처리하는가?

### 3. 기술적 정확성 ★★★★★
- [ ] **Python**: `boto3` 클라이언트가 `lru_cache` 싱글턴을 사용하는가?
- [ ] **Python**: `except Exception` 대신 구체적인 예외(`ClientError` 등)를 처리하는가?
- [ ] **TypeScript**: `any` 타입을 사용하지 않았는가?
- [ ] **CloudFormation**: 리전/계정 ID가 하드코딩되지 않았는가?

### 4. 검증 통과 ★★★★★
- [ ] `pytest` 및 `vitest` 전체 테스트 결과가 `0 failures`인가? (회귀 없음)
- [ ] `npx tsc --noEmit` 실행 시 전체 프로젝트에 타입 에러가 없는가?
- [ ] `pylint` 복잡도 규칙(locals≤15, stmts≤50 등)을 준수하는가?

---

## ⚠️ SHOULD PASS (실패해도 스프린트는 통과 가능하나, 다음 턴에서 해결 권고)

### 5. 코드 가독성 및 구조 (★★★)
- [ ] 함수 길이가 50줄 이하인가?
- [ ] 동일한 로직이 중복되지 않고 공통 모듈(`common/`)로 추출되었는가?
- [ ] 로깅 포맷(`logger.error("msg: %s", e)`)을 준수하는가?

### 6. 테스트 품질 (★★★)
- [ ] 핵심 비즈니스 로직에 대해 단위 테스트가 작성되었는가?
- [ ] `moto` 또는 Mock을 사용하여 외부 의존성이 격리되었는가?

---

## 🚫 평가 금지 사항 (Evaluator 주의)
- "전반적으로 잘 동작하므로 PASS"와 같은 관대한 판정은 금지한다.
- Generator가 "완료했다"고 보고한 내용도 반드시 도구(Bash)를 통해 직접 확인하라.
- 사소한 스타일 차이(이미 합의된 린트 규칙 외)로 FAIL 시키지 마라.
- 대규모 리팩터링이 포함된 경우, 즉시 중단시키고 별도 스프린트로 분리하도록 지시하라.
