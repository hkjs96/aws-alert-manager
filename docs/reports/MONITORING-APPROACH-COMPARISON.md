# CloudWatch 알람 운영 방식 비교 — 1페이지 요약

> 대상: 의사결정/고객 대면용 | 두 접근의 핵심 차이와 권장 방향

## 한눈에 보기

| 구분 | **A. CFN + Metric Insights 일괄 배포** | **B. AWS Monitoring Engine (우리)** |
|------|----------------------------------------|--------------------------------------|
| 한 줄 정의 | 템플릿/스크립트로 알람을 **한 번에 배포** | 태그 기반으로 알람을 **자동 생성·동기화·정리하는 엔진 + 관리 UI** |
| 신규 리소스 커버 | 쿼리가 자동 탐지 (즉시) | 이벤트 즉시 반응 + 일 1회 보정 |
| 리소스별 임계치 | ❌ 플릿 균일값 | ✅ 태그로 리소스마다 개별 설정 (`Threshold_CPU=90`, `=off`) |
| 자동 복구 | ❌ 없음 | ✅ 무단 변경 감지 시 자동 조치 |
| 관리 UI | ❌ 콘솔 직접 | ✅ 통합 대시보드 |
| 멀티 계정 | 계정마다 수동 배포 | ✅ 중앙 허브에서 일괄 관리 |
| 배포 난이도 | ◎ 클릭 한 번 (코드 불필요) | ○ 백엔드 스택 배포 필요 |
| 운영 부하/부품 | ◎ 알람만 (compute 0) | △ Lambda/DynamoDB 등 |
| 검증 체계 | 스크립트 기반 | ✅ 435+ 자동 테스트 + 거버넌스 |

## 각 방식이 빛나는 상황

**A가 유리:** 균일 임계치로 충분한 단순 환경, 빠른 1회성 납품, 운영 인력 최소화.

**B(우리)가 유리:** 리소스별 정책이 필요하고, 실시간 대응·자동 복구·통합 콘솔·다계정 관리가 요구되는 운영 환경.

## 권장 방향 — 하이브리드

> 두 방식은 경쟁이 아니라 **결합** 가능합니다.

- **공통 메트릭(CPU 등)** → Metric Insights 쿼리로 통합 → 알람 수·운영 부하 절감, 신규 리소스 즉시 커버
- **리소스별 정책·자동 복구·UI** → 우리 엔진 유지 → A가 구조적으로 제공 못 하는 차별값

→ "균일 안전망(A) + 정밀 제어·자동 복구·콘솔(B)"을 한 제품에 담아,
   **운영 부하는 낮추고 제어력은 높이는** 포지션을 확보합니다.
